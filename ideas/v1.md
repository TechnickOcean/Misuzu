# CTF Three-Agent Workflow (v1)

## Goal
Implement a three-agent CTF solving pipeline with durable state, resumable progress, and a knowledge-augmentation loop. The system must create and bind a workspace (directory + DB record), track full solving state, and allow interruption/restart without losing context.

Agents:
- EnvAgent: prepares environment, workspace, and metadata.
- CTFAgent: primary solver, uses tools and reasoning to solve.
- AgentHiro: sub-agent for knowledge retrieval and validation.

## Scope
This document describes the implementation plan and contracts to build on the current codebase.
It does not implement the agents themselves.

## Key Requirements
1. If attachments are provided, create a `@tmp/<workspace_id or name>/` style directory (example: `tmp/example/`) and bind it to a workspace record in `src/tools/workspace/workspace.ts`.
2. If black-box (no attachments), still create a workspace and a directory, but do not populate attachments.
3. Write `Environment.md` into the workspace directory with:
   - Remote URL (if any)
   - Title, description, hints
   - Attachment list (if any)
4. Maintain a full solving state in workspace store (DB):
   - Process log
   - Current step
   - Current findings
   - Knowledge index and summaries
5. CTFAgent uses the workspace context, and calls AgentHiro when knowledge is insufficient.
6. AgentHiro performs websearch/shell/other tools to confirm knowledge, produces a reproducible report, and writes a summary index into workspace store.
7. The pipeline must be resumable: on restart, it reconstructs context from workspace store + files.

## Directory Layout
Each workspace directory follows this structure:

```
<workspace_root>/
  Environment.md
  challenge/          # attachments copied here
  knowledges/         # AgentHiro reports
  solution/           # CTFAgent artifacts
  logs/               # optional
  WriteUp.md          # final writeup (optional)
```

Existing sample: `tmp/example/` already matches this shape and should be used as template.

## Workspace Store Schema (DB)
Store a single JSON object under `workspace.store`. Suggested shape:

```
{
  "status": "idle" | "running" | "blocked" | "done",
  "current_step": "string",
  "progress": [
    { "ts": "ISO", "agent": "EnvAgent|CTFAgent|AgentHiro", "event": "string" }
  ],
  "findings": [
    { "id": "string", "summary": "string", "artifacts": ["path"], "ts": "ISO" }
  ],
  "knowledge_index": [
    { "id": "k-0001", "title": "string", "summary": "string", "source": "url|local", "path": "knowledges/<file>.md" }
  ],
  "environment": {
    "remote_url": "string|null",
    "title": "string",
    "description": "string",
    "hints": ["string"],
    "attachments": ["challenge/<file>"]
  }
}
```

This store becomes the canonical state snapshot for resuming.

## EnvAgent

### Responsibilities
- Create workspace record (`createWorkspaceTool`) with a descriptive title.
- Resolve workspace path from DB (`getDBWorkspace`).
- Create directory structure.
- If attachments exist:
  - Copy to `challenge/`.
  - Record filenames in store.
- If black-box:
  - Create empty `challenge/` directory only.
- Generate `Environment.md` in workspace root.
- Initialize store with `status=idle`, `current_step=env_ready`.

### Inputs
- Title
- Description
- Hints (optional)
- Remote URL (optional)
- Attachment list (optional, file paths or file handles)

### Outputs
- Workspace ID
- Workspace path
- Initialized store
- `Environment.md`

### Environment.md Template
Use a fixed template for consistency:

```
# Environment

Title: <title>
Description: <description>
Remote URL: <url or none>

## Hints
- <hint 1>
- <hint 2>

## Attachments
- <challenge/file1>
- <challenge/file2>
```

## CTFAgent

### Responsibilities
- Load workspace store + `Environment.md`.
- Build solving context: environment, directory tree, known findings, knowledge index.
- Execute tool-assisted solving steps (shell, readFile, grep, etc).
- Append to progress log and findings.
- Maintain `current_step` and `status`.
- On success: extract final flag, write an exploit script into `solution/`, and produce a reproducible `WriteUp.md`.

### Context Assembly
When CTFAgent starts (or resumes), its prompt includes:
- Environment metadata.
- Workspace directory tree (call `globFiles` or list known paths).
- Current `findings`.
- Knowledge index summaries (id + title + summary + path).
- Current step and last progress log items.

### When to Call AgentHiro
Trigger when:
- Analysis reaches missing domain knowledge (library behavior, protocol spec, known CVE).
- The agent cannot justify next steps without external validation.

#### Recommended Call Prompt (AgentHiro)
The call should request:
- Specific unknowns to research.
- Required confidence (primary sources preferred).
- Expected output: minimal reproducible steps + links.

Example:

```
You are AgentHiro. Research the following questions with sources and validation:
1) <question>
2) <question>
Return a reproducible summary and write a report file in knowledges/.
```

### Termination Conditions
- Success: final flag obtained.
  - Persist final exploit script in `solution/`.
  - Write a reproducible `WriteUp.md` with steps and verification.
- Blocked: no valid next step without external knowledge or resources.
- Failed: repeated attempts exceed budget or environment constraints.

## AgentHiro

### Responsibilities
- Use tools: websearch, fetchMarkdown, github_search_*, shell if needed.
- Verify sources, resolve contradictions.
- Produce a reproducible report in `knowledges/`.
- Write a concise summary entry to workspace store `knowledge_index`.

### Output Format
Each knowledge report should be a Markdown file:

```
# Knowledge Report: <title>
## Summary
- ...

## Sources
- <url>

## Reproduction / Evidence
- steps

## Applicability
- when it applies
```

### Store Update
Add entry:

```
{
  "id": "k-000X",
  "title": "...",
  "summary": "...",
  "source": "url|local",
  "path": "knowledges/<file>.md"
}
```

## Resume/Interrupt Behavior
- Every agent writes progress updates to the store after each meaningful step.
- `status` is set to `blocked` if waiting on AgentHiro.
- On restart, CTFAgent reads store and continues from `current_step`.

## Implementation Plan

### 1) Add EnvAgent
Create `src/agents/EnvAgent.ts`.
- Provide a function to initialize workspace, copy attachments, and write `Environment.md`.
- Use `createWorkspaceTool` + filesystem ops (via bun file APIs or Node).
- Update workspace store via `manageState` or direct DB call.

### 2) Extend Workspace Utilities
Add helper functions for:
- `ensureWorkspaceLayout(path)`
- `writeEnvironmentMd(path, data)`
- `appendProgress(workspace_id, event)`
- `updateStatus(workspace_id, status, step)`

### 3) Implement CTFAgent
Create `src/agents/CTFAgent.ts` with a ToolLoop-based strategy.
- Load workspace
- Build context
- Loop until solved or blocked

### 4) Implement AgentHiro
`src/agents/AgentHiro.ts` should be upgraded from placeholder:
- Accept a knowledge request
- Use tools in `src/tools/websearch` + `src/tools/workspace`
- Produce report and update store

### 5) Orchestrator
Create `src/agents/CTFOrchestrator.ts` to:
- Start EnvAgent
- Start CTFAgent
- Dispatch AgentHiro on demand
- Monitor and persist status

## Tooling Implications
- `workspace.ts` already supports file, shell, and state operations.
- `GitHubSearch` provides code/repo/advisory search.
- `WebSearch` provides general search + markdown conversion.

## Failure Handling
- Any tool failure should return a structured error (`AppError`) and be logged by pino.
- Agent should treat `UPSTREAM_ERROR` as a retryable or alternative-path trigger.

## Deliverables
- `src/agents/EnvAgent.ts`
- `src/agents/CTFAgent.ts`
- `src/agents/AgentHiro.ts` (implemented)
- `src/agents/CTFOrchestrator.ts`
- Workspace helper utilities
- `ideas/v1.md` (this document)

## Improvements Based on Recent Research/Practice
- Explicit stop conditions to prevent unbounded tool loops and wasted budget.
- Short reflection step after each major attempt to update plan and avoid repeating failed paths.
- Evidence standard in knowledge reports (source quality + confidence level).
- Require reproducible outputs (commands/PoC/verification steps) instead of pure narrative.
- Add a tool budget counter in store and enforce fallback strategies when exceeded.
- Deduplicate knowledge queries by checking the knowledge index before dispatching AgentHiro.
