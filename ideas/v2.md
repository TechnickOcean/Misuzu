# CTF Agent System V2: Stateful & Observable Architecture

## Core Philosophy

Transition from "script-based" execution to "state-machine" execution. The agent must be pause-able, resume-able, and observable at every step. The context (LLM memory) is a first-class citizen that must be persisted alongside the workspace state.

## 1. Architecture Overhaul

### 1.1 State Persistence (The "Brain" on Disk)
The `workspaces` table in DB is insufficient for high-fidelity context restoration.
We will introduce a dedicated `AgentState` structure stored in the workspace's file system (e.g., `.misuzu/agent_state.json`) or a new DB column, to decouple "business logic state" (findings, knowledge) from "LLM execution state" (chat history, token usage).

**Proposed Schema Extension:**
- **`store` (Business State):** Findings, Knowledge, Progress (Existing).
- **`context` (Execution State):** Full `ChatCompletionMessageParam[]` history.
- **`metadata` (Control State):** `step_count`, `last_active_agent`, `status` (running/paused/error).

### 1.2 The "Tick" Loop
The `ToolLoopAgent` currently runs a `while` loop internally. This blocks the caller.
**Refactor:** The loop should be inverted or exposed via an Async Generator (`async function* run()`) or an Event Emitter pattern.
- **Yields:** `StepStart`, `ToolCall`, `ToolResult`, `ModelResponse`, `StepEnd`.
- **Allows:** The frontend to consume events in real-time (SSE/WebSocket).

## 2. Prompt Engineering & Context Management (Crucial)

The current prompts are too generic ("You are CTFAgent..."). We need a **Structured Thinking Process** to enforce tool usage discipline.

### 2.1 The "OODA Loop" Prompt Strategy
Enforce a cognitive cycle in the system prompt: **Observe -> Orient -> Decide -> Act**.

**New System Prompt Structure:**
```markdown
You are CTFAgent, an autonomous cyber security expert.
Your goal is to solve the challenge in this workspace.

## STRICT OPERATIONAL PROTOCOL
1. **OBSERVE**: Read the output of previous tools. Do not hallucinate file contents.
2. **ORIENT**: Update your mental model. What do we know? What is missing?
3. **DECIDE**: Formulate a hypothesis or a specific check.
4. **ACT**: Execute ONE set of tools to verify the hypothesis.

## CONSTRAINTS
- **No Blind Guessing**: Verify file existence (glob/ls) before reading.
- **Atomic Actions**: Do not chain 5 commands in one shell call unless necessary.
- **Validation**: After running an exploit, YOU MUST verify if it worked (e.g., check for flag).
- **Escalation**: If stuck for 3 steps, use `ask_hiro` to get external knowledge.
```

### 2.2 Dynamic Context Windowing
Instead of dumping `fileTree` and `envMd` raw into the prompt every time (which wastes tokens and confuses attention):
- **File Tree:** Only show top-level and depth=2. If deeper, agent must `ls`.
- **Findings:** Summarize findings into a "Current Intelligence" block.
- **Output Truncation:** The `2000` char limit is good, but we need a "smart tail" or "grep" suggestion in the prompt when truncation happens.

## 3. Tool Definition Improvements

The LLM struggles to know *when* to use which tool. We will sharpen the definitions.

- **`shell`**: Explicitly warn against interactive commands (`vim`, `top`, `nc -l`). Suggest `timeout` usage.
- **`readFile`**: Add a `purpose` parameter (optional, string) to force the LLM to "think" about why it needs this file (CoT in arguments).
- **`ask_hiro`**: Renamed from generic "Hiro" call. Explicitly defined as "Search/Research/Knowledge Retrieval".

## 4. Web/Frontend Integration Plan

### 4.1 API Interface
- `POST /api/agent/start`: Spawns the loop.
- `GET /api/agent/events`: SSE stream of the agent's "Tick" loop.
- `POST /api/agent/stop`: Sets a `stop_signal` flag in the DB/State.
- `POST /api/agent/input`: Sends human feedback (if agent requests intervention).

### 4.2 Recovery Flow
1. **Load:** API reads `agent_state.json`.
2. **Restore:** `new ToolLoopAgent({ initialContext: state.history })`.
3. **Resume:** Continue the loop from `state.step_count`.

## 5. Implementation Roadmap (V2)

1. **Refactor `ToolLoopAgent`**:
   - Make it an `AsyncGenerator`.
   - Add `saveState()` hook.
   - Support `initialContext` hydration.

2. **Refactor `CTFAgent`**:
   - Implement the "OODA" System Prompt.
   - Bind the State Manager (DB <-> File System).
   - Expose the Event Stream.

3. **Enhance Tools**:
   - Add `purpose` field to critical tools.
   - Improve `shell` safety wrapper (timeout injection).

4. **Integration Test**:
   - Simulate a "kill -9" (restart server) and verify the agent resumes exactly where it left off.
